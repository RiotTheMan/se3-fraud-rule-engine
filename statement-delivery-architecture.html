Solution Architecture

Secure File Statement Delivery

A system for storing customer account statements as PDF files and delivering them via secure, time-limited download links. This document describes the architecture, security model, data design, and implementation approach — technology choices are given as recommendations, not mandates.

Problem: Secure PDF storage + time-limited customer access
Core concerns: Confidentiality · Integrity · Non-repudiation · Availability
Recommended stack: Any REST framework · PostgreSQL · S3-compatible object store · Message queue
1 — Problem Framing

What we need to store

Customer account statements as PDF files. These are sensitive financial documents — they must never be publicly readable, must not be accessible by other customers, and must remain available for the duration of their retention period.

What we need to deliver

A way to give a specific customer temporary access to their specific statement. The access must expire automatically, must not be reusable, and must be auditable — we need to know exactly who downloaded what and when.

Why this is hard

A direct file URL is permanent and shareable. Embedding auth tokens in URLs leaks them to browser history and server logs. Generating PDFs synchronously blocks the API. Without an audit log, you can't prove or disprove a claim that a statement was accessed.

The central security insight

Treat the download URL like a one-time password. It grants access to exactly one resource for a short window. After that window closes — or after it's used once — it's permanently void. No exceptions.

2 — High-Level Architecture
External Actors
🧑‍💼

Customer

Requests statements via a portal or mobile app. Clicks a link to download their PDF. Has no direct access to the file store.

⚙️

Internal System / Scheduler

Triggers statement generation (e.g. end-of-month batch job or event from the core banking system).

📧

Notification Channel

Email or SMS service that delivers the download link to the customer once the PDF is ready.

API & Auth Layer
🛡️

Identity Provider (IdP)

Issues signed JWT tokens to authenticated callers. The Statement Service validates every inbound request against this token.

  • Any standard OAuth 2.0 / OIDC provider (Keycloak, Auth0, Cognito, etc.)
  • Role-based access: StatementRead and StatementWrite
  • Download endpoint is token-gated differently — see section 4
📋

Statement Service (API)

The single backend service that owns the full statement lifecycle. Customers and internal systems only ever talk to this service.

  • REST API on port 8080
  • Validates JWTs, enforces ownership, issues tokens
  • Never exposes S3 paths or credentials to callers
  • Publishes generation requests to the message queue
Async Processing
📨

Message Queue

Decouples PDF generation from the API request. The API publishes a job; the worker consumes it independently.

  • Any durable queue: Kafka, RabbitMQ, SQS, etc.
  • At-least-once delivery — worker must be idempotent
  • Three event types: requested, generated, delivered
📄

PDF Generator Worker

Consumes generation jobs, fetches the customer's transaction data, renders the PDF, and uploads it to object storage.

  • Can be a separate service or a worker thread in the same process
  • Must be idempotent — check if the PDF already exists before generating
  • Publishes a "generated" event on completion
🪣

Object Storage

Stores the PDF files. Entirely private — no public access policy. Customers never receive an S3 URL directly.

  • AWS S3, GCS, Azure Blob, or self-hosted MinIO
  • Encryption at rest (SSE)
  • Retention lifecycle policy (e.g. archive after 7 years)
  • Path is an opaque UUID — not guessable
Data Layer
🗄️

Relational Database

Stores statement metadata, download tokens (as hashes), and the audit log. PostgreSQL recommended.

  • Three tables: statement_requests, download_tokens, audit_log
  • Schema managed by a migration tool (Flyway, Liquibase, etc.)
  • Tokens stored as SHA-256 hashes only — never the raw value
🔒

Secrets Manager

Stores credentials at runtime. The application reads them on startup — they are never in source code or config files.

  • HashiCorp Vault, AWS Secrets Manager, or env vars in dev
  • Secrets: DB password, S3 credentials, SMTP password
📊

Observability

Health checks, structured logging, and metrics for the running service.

  • Health endpoint: /health
  • Structured JSON logs — never log raw tokens
  • Metrics: token issuance rate, download count, generation latency

4 — Token Security Design
Why not a presigned S3 URL? Presigned S3 URLs work, but they embed the bucket name and object key — which reveals your storage layout and ties your security model to your storage provider. A service-issued token keeps all that knowledge inside your service and lets you add ownership checks, rate limiting, and audit logging that S3 can't provide.
1

Generate a cryptographically random token

Use a cryptographically secure random number generator (SecureRandom in Java, secrets module in Python, crypto.randomBytes in Node.js) to produce 32 bytes. Encode as URL-safe Base64. This gives you 256 bits of entropy — brute-force is computationally infeasible.

2

Store only the SHA-256 hash

Compute SHA-256(rawToken) and store that in the database. The raw token goes into the URL only. Even if someone dumps your database, they cannot reverse a SHA-256 hash to obtain a working token. This is the same principle behind secure password storage.

3

Set a short expiry (TTL)

Default: 15 minutes. Long enough for a customer to click the link. Short enough that a leaked URL becomes worthless almost immediately. Store expires_at as an absolute timestamp, not a duration. Respond with HTTP 410 Gone (not 404) so the client knows the resource existed but the window has closed.

4

Enforce single-use atomically

Inside a single database transaction: check used = false AND expires_at > now(), then immediately set used = true. Both operations must be in the same transaction. If two requests arrive concurrently for the same token, one wins and the other gets a 410. No second chances.

5

Never log the raw token

Log the internal token row ID only (a database integer). The raw token must not appear in application logs, access logs, or error reports. If it does, treat it as a security incident. Browser history and server logs are considered untrusted surfaces.

6

Limit active tokens per statement

Allow at most 3 active (unused + unexpired) tokens per statement at any time. When a new token is requested, revoke the previous ones. This prevents an attacker from generating a large number of tokens to extend their effective access window. Log excess issuance attempts to the audit log.

Do not: store the raw token in the DB, log it anywhere, use sequential IDs or UUIDs as tokens, expose the S3 object key in the URL, or serve the download endpoint from a CDN (CDNs cache by URL — a cached download means a reused token).
Do: use a CSPRNG, store hash only, enforce single-use atomically, keep TTL short, serve via HTTPS only, record every redemption attempt (successful or not) in the audit log with the client IP address.

POST /api/v1/statements Write role Request statement generation for a customer and date range. Body must include an idempotencyKey so retries don't create duplicates. Publishes an async job and returns immediately with status REQUESTED. 201 — statement summary

GET /api/v1/statements/download Token = credential. No JWT required. Redeem a download token. Query param: ?token=…. Validates hash, expiry, and single-use. Streams the PDF. Designed for direct browser navigation — the customer just clicks the link. 200 — PDF stream
410 — expired / used
404 — unknown
GET /api/v1/statements/{id}/audit Write role Full immutable audit trail for a statement. For compliance and back-office investigation. 200 — audit event list
Idempotency on POST /statements: the caller supplies an idempotencyKey (e.g. a UUID generated client-side). If the same key arrives twice, return the existing record rather than creating a duplicate. Store it with a UNIQUE constraint — the database enforces it even under concurrent load.
Ownership enforcement: before issuing a download token, the service must verify that the authenticated caller is the customer on the statement — not just that they have the Read role. A back-office user with the Write role can bypass this check for support purposes.

Idempotency in the PDF worker: the queue guarantees at-least-once delivery, which means the worker may receive the same event twice. Before generating, the worker must check whether the PDF already exists in object storage (by looking up the statementId or checking the statement_requests.status). If it does, skip generation and re-publish the statement.generated event.
8 — Security Controls
ThreatControlHow it works
Unauthenticated API access JWT validation Every endpoint (except /download) requires a valid JWT from the IdP. Expired or tampered tokens are rejected.
Customer A reads Customer B's statement Ownership check Before issuing any token, the service compares the authenticated user's ID to the customer_id on the statement. Mismatch → 403 Forbidden.
Brute-forcing a download token 256-bit random token 32 bytes from a CSPRNG. 2²⁵⁶ possible values. Statistically impossible to guess.
Intercepted token reused Single-use enforcement Atomic database transaction: mark used = true before streaming. A second request for the same token gets 410 Gone.
Stale or forwarded link 15-minute TTL Hard expiry stored as an absolute timestamp. No renewal. Expired tokens return 410, not 404.
Database breach exposes tokens Hash-only storage Only SHA-256(rawToken) is persisted. SHA-256 is a one-way function — the raw token cannot be recovered from the hash.
Direct access to stored files Private object storage Bucket has no public access policy. Storage keys are opaque UUIDs. Customers never receive a storage URL — only the service-issued token URL.
Credentials in source code Secrets manager All credentials (DB password, storage access keys) are read from a secrets manager at startup — never hardcoded or in config files committed to version control.
Token hoarding attack Active token limit Max 3 active tokens per statement. New issuance revokes all previous ones. Prevents accumulating tokens to extend the access window.
No forensic trail after an incident Immutable audit log Every event (token issued, download attempt, expiry) is appended to audit_log with timestamp and client IP. The log is never updated or deleted.

10 — Common Mistakes to Avoid

Serving PDFs directly from object storage

Making the S3 bucket public (or using long-lived presigned URLs) means anyone who gets hold of the URL can access the file indefinitely. Always proxy through your service so you can enforce expiry and single-use.

Using a UUID or sequential ID as the download token

UUIDs are guessable if an attacker knows the format. Sequential IDs are trivially enumerable. Use a cryptographically random value — the difference between a guessable token and an unguessable one is the difference between security and the appearance of security.

Checking expiry without enforcing single-use

A token that expires in 15 minutes but can be used unlimited times within that window is a much weaker control. If someone intercepts the link (browser history, email forward, Slack message), they can download the file. Single-use + expiry together close both attack vectors.

Generating PDFs synchronously in the API request

PDF generation can take seconds, especially for long statement periods. Blocking the HTTP request for that long degrades user experience and makes the service fragile under load. Always generate asynchronously and notify the customer when the file is ready.

No idempotency key on generation requests

Without an idempotency key, a retry (from a network blip, a double-click, or a scheduler bug) creates a duplicate statement. The client must supply a unique key per logical request, and the server must deduplicate against it.

Skipping the audit log

Financial statements are regulated documents. Without an audit trail, you cannot answer "did this customer download their statement?" or "was this file accessed without authorisation?". The audit log is not optional — it is the evidence record for disputes and compliance checks.
